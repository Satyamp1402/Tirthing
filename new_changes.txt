CHANGE 1: AUTHENTICATION MIGRATION (localStorage → Cookies)
================================================================================

WHAT CHANGED:
  Before: When a user logged in, the backend sent a token in the response body.
          The frontend saved it in localStorage and attached it to every request.
  After:  The backend now sets the token as an httpOnly cookie. The browser
          handles everything automatically — the frontend never touches the token.

WHY:
  localStorage is readable by any script on the page. If a hacker injects
  malicious JavaScript (called XSS), they could steal the token and pretend
  to be the user. httpOnly cookies are invisible to JavaScript entirely —
  even if a hacker injects a script, they CANNOT access the token.

FILES CHANGED:

  BACKEND:
  --------
  auth.controller.js
    - Added sendTokenCookie() function that puts the JWT in a cookie
    - Added COOKIE_OPTIONS with httpOnly, secure, sameSite settings
    - Added getMe() function — frontend calls this to check "am I logged in?"
    - Added logout() function — clears the cookie

  auth.middleware.js
    - Changed from: reading token from "Authorization" header
    - Changed to: reading token from req.cookies.token

  auth.routes.js
    - Added GET /me route (session validation)
    - Added POST /logout route (cookie clearing)

  server.js
    - Added cookie-parser middleware
    - Added credentials: true to CORS config

  FRONTEND:
  ---------
  context/AuthContext.jsx (NEW FILE)
    - Created a React Context that manages login state for the entire app
    - On page load, calls /api/auth/me to check if the cookie is valid
    - Provides user, isLoading, login(), logout() to all components

  api/axios.js
    - Removed the old token interceptor
    - Added withCredentials: true (browser sends cookies automatically)

  Login.jsx
    - Removed localStorage.setItem('token')
    - Now calls login(data.user) from AuthContext

  Signup.jsx
    - Same changes as Login.jsx

  Navbar.jsx
    - Changed from localStorage.getItem('user') to useAuth()

  Sidebar.jsx
    - Changed from localStorage.getItem('user') to useAuth()

  ProtectedRoute.jsx
    - Changed from checking localStorage to checking AuthContext
    - Added loading state to prevent flash-to-login bug

  main.jsx
    - Wrapped entire app in AuthProvider

CHANGE 2: useTransition in GenerateItinerary
================================================================================

FILE: pages/user/GenerateItinerary.jsx

WHAT CHANGED:
  Before: Used useState for loading → [loading, setLoading]
  After:  Used useTransition → [isPending, startTransition]

WHY:
  useTransition is a React 19 feature. It marks the API call as "non-urgent"
  so React keeps the form responsive while waiting for the response.
  The user can still edit form fields during generation.

CHANGE 7: Accessibility Mode Toggle
================================================================================

NEW FILE: hooks/useAccessibility.js
UPDATED: components/Sidebar.jsx
UPDATED: index.css

WHAT IT DOES:
  - "A11y: Off" button in the sidebar footer
  - When toggled ON:
    → Text gets bigger (adds "text-lg" to HTML element)
    → All animations and transitions are disabled
    → Buttons get more padding (bigger touch targets)
  - Preference saved in localStorage so it survives page refreshes

WHY:
  Designed for elderly pilgrim users (65+) who may find small text
  hard to read and rapid animations disorienting.

CHANGE 9: Transport Cost Display
================================================================================

UPDATED: backend/utils/engine.js
UPDATED: pages/user/ItineraryResult.jsx

WHAT CHANGED:

  Backend (engine.js):
    Before: The engine calculated transport costs internally but only saved
            a total per day. Individual per-hop data was thrown away.
    After:  Each place now includes:
              transportMode  → "walk", "cycle_rickshaw", or "auto_rickshaw"
              transportCost  → fare in rupees (0 for walking)
              distanceKm     → road distance between sites
              travelTimeMin  → estimated travel time in minutes

  Frontend (ItineraryResult.jsx):
    Between each pair of sites in the timeline, shows:
      🚶 Walk 0.8 km                          (if < 1.5 km)
      🛺 Rickshaw 3.2 km • ~12 min • ₹68     (if < 5 km)
      🛺 Auto 7.5 km • ~25 min • ₹153        (if > 5 km)

  BUG FIX:
    This was a data-shape mismatch we caught during the audit. The frontend
    expected transport data on each place object, but the engine wasn't
    providing it. The engine was calculating it correctly but discarding the
    per-hop values. We fixed the engine to embed this data.

CHANGE 10: Delete Itinerary API
================================================================================

NEW: Backend DELETE endpoint for itineraries

FILES:
  - controllers/itinerary.controller.js → Added deleteItinerary function
  - routes/itinerary.routes.js → Added DELETE /:id route
  - services/itinerary.service.js → Added deleteItinerary() method

SECURITY:
  The delete function checks ownership — a user can ONLY delete their own
  itineraries. If user A tries to delete user B's trip, they get a 403
  (Forbidden) error. This prevents users from deleting each other's data.
