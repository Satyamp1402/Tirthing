1. REACT 19 — WHY IT MATTERS
==============================

React 19 introduced new "hooks" (special functions) that make apps feel
faster and more responsive. We use three of them:

  useTransition    → Keeps forms responsive during slow API calls
  useDeferredValue → Keeps search inputs fast while filtering large lists
  useOptimistic    → Makes deletions feel instant

These are NOT just fancy additions — they solve REAL user experience problems
that existed with the old approach.


2. useTransition — IN GENERATE ITINERARY
==========================================

FILE: pages/user/GenerateItinerary.jsx

THE PROBLEM (old way):
  When the user clicks "Generate Itinerary", the backend engine takes 2-5
  seconds to compute the plan. With the old approach:
    const [loading, setLoading] = useState(false)
    setLoading(true)
    await api.post('/itinerary/generate', data)
    setLoading(false)
  
  During those 2-5 seconds, the ENTIRE form freezes. The user can't:
  - Edit the budget field if they made a typo
  - Click "Cancel" if they changed their mind
  - Scroll the page
  React is busy re-rendering with loading=true and blocks everything.

THE SOLUTION (useTransition):
  const [isPending, startTransition] = useTransition()
  
  startTransition(async () => {
    await api.post('/itinerary/generate', data)
  })

  startTransition tells React: "This update is NOT urgent. Keep the UI
  responsive while you work on it in the background."

  During those 2-5 seconds:
  - The form stays interactive — user CAN edit fields
  - isPending becomes true (we show "Generating..." on the button)
  - React processes the API call without freezing the rest of the page
  - When the call finishes, isPending becomes false automatically

ANALOGY:
  Old way: You order food at a restaurant and STAND at the counter,
  unable to do anything else until it arrives.
  New way: You order food and SIT DOWN at your table. You can browse the
  menu, chat, or even change your order. The food arrives when ready.


3. useDeferredValue — IN BROWSE PLACES
========================================

FILE: pages/user/BrowsePlaces.jsx

THE PROBLEM:
  We have 266+ places in the database. When the user types in the search
  box, React needs to:
    1. Update the input text (what the user sees)
    2. Filter all 266 places to find matches
    3. Re-render the grid of matching cards

  With the old approach, steps 1-3 happen together. If filtering 266 places
  takes 50ms, the input field lags — you type "V" and there's a noticeable
  delay before it appears. This feels terrible on mobile devices.

THE SOLUTION (useDeferredValue):
  const [searchInput, setSearchInput] = useState('')
  const deferredSearch = useDeferredValue(searchInput)

  - searchInput updates IMMEDIATELY when the user types (step 1)
  - deferredSearch updates a moment LATER (steps 2-3 happen after)
  - We filter places using deferredSearch, not searchInput
  - While deferredSearch catches up, we dim the list slightly (opacity 0.7)

  This means the keyboard ALWAYS responds instantly. The list re-filters
  a fraction of a second later. The user never feels lag.

ANALOGY:
  Imagine you're dictating a letter to someone. 
  Old way: They write each word as you say it, making you WAIT between words.
  New way: You speak freely at your pace. They write a moment behind you,
  catching up. You never have to slow down or pause.


4. useOptimistic — IN MY ITINERARIES
======================================

FILE: pages/user/MyItineraries.jsx

THE PROBLEM:
  When the user deletes a trip card:
    Old way:
      1. Click trash icon
      2. Show spinner on the card
      3. Wait for DELETE API to complete (200-500ms)
      4. If success → remove card from list
      5. If fail → show error
    
    The user stares at the card for 500ms wondering "did it work?"

THE SOLUTION (useOptimistic):
  const [optimisticTrips, setOptimistic] = useOptimistic(
    trips,
    (currentTrips, deletedId) => currentTrips.filter(t => t._id !== deletedId)
  )

  1. Click trash icon
  2. Card disappears INSTANTLY (React optimistically removes it from the list)
  3. API DELETE call runs in the background
  4. If success → great, the card stays gone
  5. If fail → React AUTOMATICALLY puts the card back + shows error toast

  The "optimistic" part means we ASSUME the operation will succeed and
  update the UI immediately. If we're wrong, we roll back.

ANALOGY:
  Old way: You ask your friend "can I borrow your pen?" and WAIT for them
  to say yes before picking it up.
  New way: You pick up the pen immediately (optimistically assuming yes).
  If they say "no", you put it back.


5. SKELETON LOADING SCREENS
=============================

FILE: components/common/SkeletonCard.jsx

WHAT ARE SKELETONS?
  Gray placeholder shapes that mirror the EXACT layout of real content.
  They show BEFORE the real data arrives from the API.

THREE VARIANTS:
  "card" → Used in BrowsePlaces
    Shows: gray rectangle (image) + two gray lines (title, subtitle)
  
  "stat" → Used in Dashboard
    Shows: gray circle (icon) + gray lines (number, label)
  
  "trip" → Used in MyItineraries
    Shows: gray lines mimicking a trip card with destination and details

WHY SKELETONS INSTEAD OF SPINNERS?
  A spinner (loading circle) just says "something is loading."
  A skeleton says "here's EXACTLY what shape the content will be."

  Benefits:
  1. Sets expectations — user knows cards are coming, not a single paragraph
  2. Prevents layout shift — when real data arrives, it fills the SAME space
  3. Feels faster — studies show users perceive skeleton-loaded pages as
     loading 15-20% faster than spinner-loaded pages (same actual time)

THE ANIMATION:
  Each skeleton has a "pulse" animation — it subtly fades in and out to
  indicate loading. When accessibility mode is on, this animation stops
  (some elderly users find it disorienting).


6. ACCESSIBILITY MODE (a11y)
==============================

FILE: hooks/useAccessibility.js

WHO IS THIS FOR?
  Our target users include elderly pilgrims (65+ years old) who may have:
  - Poor eyesight → small text is hard to read
  - Motor difficulties → small buttons are hard to tap on mobile
  - Cognitive sensitivity → rapid animations are disorienting

HOW IT WORKS:
  Toggle in the Sidebar: "A11y: Off" → click → "A11y: On"

  When ON, three things change:
    1. TEXT SIZE: Adds "text-lg" CSS class to the <html> element
       → All text gets about 15% larger across the entire app

    2. ANIMATIONS: Sets data-reduce-motion="true" on <html>
       → CSS rule in index.css responds:
         html[data-reduce-motion="true"] * {
           animation-duration: 0.01ms !important;
           transition-duration: 0.01ms !important;
         }
       → ALL animations and transitions are effectively disabled
       → Hover effects, loading pulses, page transitions — all stop

    3. BUTTON PADDING: Sets a CSS variable for larger touch targets
       → Buttons get more padding → easier to tap on mobile

  PERSISTENCE: Saved in localStorage('a11y_mode')
    → If the user refreshes, their preference is remembered
    → This is one of the few appropriate uses of localStorage
      (it's not sensitive data, just a preference)


7. BUDGET TRACKER
==================

FILE: pages/user/BudgetTracker.jsx
ROUTE: /trip/:itineraryId/tracker

WHAT IT DOES:
  A personal expense tracker for each trip. After generating an itinerary,
  the user can track their actual spending vs planned budget.

FEATURES:
  - Shows planned budget (from the itinerary)
  - "Add Expense" form: category, amount, note
  - Progress bar that changes color:
      Green  → Under 90% of budget (you're fine)
      Yellow → 90-100% of budget (warning — running low)
      Red    → Over 100% (over budget!)
  - Category breakdown: How much spent on each category
  - Delete expenses (click the × button)
  - Total remaining budget display

7 EXPENSE CATEGORIES:
  🚗 Transport | 🍽 Food | 🏨 Stay | 🎫 Entry Fee |
  🙏 Offering | 🛍 Shopping | 📦 Other

WHY localStorage?
  1. Works OFFLINE — pilgrims may be at remote temples with no internet
  2. Instant saves — no API call needed, no loading time
  3. Personal data — expenses don't need to be shared or synced
  4. No backend changes required — simple and reliable

DATA STRUCTURE:
  Key: budget_expenses_{itineraryId}
  Value: JSON array of expense objects
    [
      { id: "abc123", category: "food", amount: 250, note: "Lunch", date: "..." },
      { id: "def456", category: "transport", amount: 80, note: "Rickshaw", date: "..." }
    ]


8. WHATSAPP SHARE
==================

FILE: pages/user/ItineraryResult.jsx

WHY:
  Indian pilgrims use WhatsApp as their primary communication channel.
  Families share trip plans in WhatsApp groups to coordinate travel,
  pickups, and logistics. A share button is essential.

HOW:
  1. Generate a plain text summary of the itinerary (under 1000 characters)
  2. Open WhatsApp with the text pre-filled:
     window.open(`https://wa.me/?text=${encodeURIComponent(text)}`)
  3. On mobile: opens WhatsApp directly
  4. On desktop: opens WhatsApp Web

  The 1000-character limit ensures the message displays well in chat bubbles
  without being cut off or overwhelming.


9. OFFLINE SUPPORT
===================

TWO MECHANISMS:

A) useNetworkStatus hook:
  Listens to browser's "online" and "offline" events.
  Returns true/false for current network state.
  
  In App.jsx, if offline:
    /generate-itinerary → Shows offline message (can't generate without API)
    /my-itineraries → Shows OfflineItineraries page (reads from localStorage)

B) Offline Itineraries (pages/OfflineItineraries.jsx):
  Scans localStorage for all keys starting with "saved_itinerary_"
  Displays cached trips in a simple card format.
  Users can click to view the full offline itinerary.

C) Itinerary Caching (in ItineraryResult.jsx):
  When you view any itinerary, it saves a copy to localStorage.
  If you lose internet later, the offline page can display it.

12. THE DESIGN SYSTEM (index.css)
==================================

All colors follow a warm saffron pilgrimage theme:

  Backgrounds:   Saffron cream (#FFF4D6) — warm, inviting, not cold
  Surface:       Saffron ivory (#FFF9E6) — cards and panels
  Primary color: Saffron orange (#FB923C) — buttons, links, accents
  Text:          Dark warm brown (#32261E) — high contrast, readable
  Borders:       Semi-transparent warm (#B47828/25%) — subtle, not harsh
  Gradient:      Saffron → deep orange — for primary buttons

WHY SAFFRON?
  Saffron is the color of Hindu spirituality and pilgrimage. It immediately
  communicates the app's purpose. Using a generic blue/gray theme would
  feel corporate and disconnected from the target audience.

TAILWIND CSS v4 SPECIFICS:
  - NO tailwind.config.js file — all configuration is in CSS
  - Theme is defined inside @theme {} block in index.css
  - Custom colors use CSS custom properties (--color-primary, etc.)
  - This is Tailwind v4's new architecture, different from v3

LIKELY VIVA QUESTIONS AND ANSWERS
================================================================================

Q: What is useTransition and how is it different from a loading state?
A: useState loading blocks the UI — the entire component re-renders in 
   "loading" mode and nothing is interactive. useTransition marks the update 
   as "non-urgent" — React keeps the form interactive while processing the 
   slow operation in the background.

Q: What is useDeferredValue? Give a simple example.
A: It creates a "delayed copy" of a value. The original updates instantly 
   (keeping the input responsive), while the deferred copy updates a moment 
   later (triggering the expensive filtering). It's like live captioning — 
   the speaker talks freely, and the text appears slightly behind.

Q: What is useOptimistic? What happens if the API call fails?
A: useOptimistic updates the UI immediately, assuming the operation will 
   succeed. If the API fails, React automatically reverts to the previous 
   state. The user never notices the brief optimistic state — they just see 
   the card reappear with an error message.

Q: Why skeleton screens instead of spinners?
A: Skeletons show the SHAPE of content that's coming, setting expectations 
   and preventing layout shifts. Studies show users perceive skeleton-loaded 
   pages as 15-20% faster than spinner-loaded pages.

Q: Why did you build an accessibility mode?
A: Our primary users include elderly pilgrims (65+). Many have poor eyesight, 
   difficulty with small touch targets, and sensitivity to rapid animations. 
   The toggle increases text size, enlarges buttons, and disables animations.

Q: How does offline support work without a service worker?
A: We save itinerary data to localStorage when the user views it. If they 
   go offline later, the OfflineItineraries page scans localStorage for 
   cached trips. The budget tracker also uses localStorage — fully offline.
   (Note: We DO have a service worker via the PWA plugin for asset caching.)

Q: Why store budget expenses in localStorage instead of the database?
A: Three reasons: (1) Works offline at remote pilgrimage sites. (2) Expenses 
   are personal and don't need syncing. (3) No backend changes needed.

Q: What is a Progressive Web App (PWA)?
A: A PWA is a web app that can be installed on a phone like a native app. 
   It works offline (cached assets), has a home screen icon, and loads fast.
   We use vite-plugin-pwa to generate a service worker automatically.

Q: What is the difference between Tailwind v3 and v4?
A: Tailwind v3 uses a tailwind.config.js JavaScript file for configuration. 
   Tailwind v4 moves ALL configuration into CSS, using @theme {} blocks 
   and @variant for variants like dark mode. No JS config file needed.

Q: What is React Router's Outlet?
A: Outlet is a placeholder that renders the matching child route. The 
   DashboardLayout has a sidebar + header that stay constant, and the 
   Outlet area shows whichever page matches the current URL.

Q: How does WhatsApp sharing work technically?
A: We generate a plain text summary, URL-encode it, and open 
   https://wa.me/?text=encodedText. WhatsApp's deep link system handles 
   the rest — opening the app on mobile or WhatsApp Web on desktop.
