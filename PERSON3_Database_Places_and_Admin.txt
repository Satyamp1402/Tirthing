1. MONGODB — THE DATABASE
==========================

MongoDB is a "NoSQL" database. Unlike traditional SQL databases (tables with
rows and columns), MongoDB stores data as "documents" — which look like
JavaScript objects.

Example of a place document in MongoDB:
  {
    _id: "64abc...",
    name: "Kashi Vishwanath Temple",
    city: "Varanasi",
    image: "https://res.cloudinary.com/...",
    latitude: 25.3109,
    longitude: 83.0107,
    visitDuration: 1.5,
    entryFee: 0,
    priority: 1,
    description: "One of the most famous Hindu temples...",
    time_slot: "morning",
    theme_group: "temples",
    createdAt: "2024-03-15T10:30:00Z"
  }

MONGOOSE is the library we use to interact with MongoDB from Node.js.
It provides "schemas" — blueprints that define what a document should look like
and validates data before saving.


2. THE PLACE MODEL (models/place.model.js)
===========================================

This defines the structure of every place/site in our database:

  BASIC INFORMATION:
  - name         → "Kashi Vishwanath Temple" (required)
  - city         → "Varanasi" (required) — used to filter places by destination
  - description  → "One of the twelve jyotirlingas..." (optional)

  IMAGE:
  - image        → Cloudinary URL where the photo is hosted (required)
  - publicId     → Cloudinary's internal ID for this image (required)
                    Used later to delete the image when the place is removed

  GEOGRAPHIC COORDINATES (for the engine):
  - latitude     → 25.3109 (north-south position)
  - longitude    → 83.0107 (east-west position)
  These are used by the engine to calculate distances between places.

  PLANNING DATA (for the engine):
  - visitDuration → 1.5 hours (how long a pilgrim should spend here)
  - entryFee      → 0 or 50 (rupees per person)
  - priority      → 1, 2, or 3
      1 = Must Visit (like Kashi Vishwanath — every pilgrim goes here)
      2 = Recommended (like Sarnath — worth visiting if time allows)
      3 = Optional (like a small local shrine — only if extra time)

  SCHEDULING HINTS (for the engine):
  - time_slot:
      'dawn'    → 5:00 AM sites (sunrise Aarti at the ghats)
      'morning' → 7:00 AM - 1:00 PM (temple visiting hours)
      'aarti'   → 6:30 PM (sunset ceremony — fixed time)
      'anytime' → Flexible, can visit whenever

  - theme_group:
      'ghats'   → Riverside steps (Dashashwamedh, Assi, Manikarnika)
      'temples' → Religious buildings (Kashi Vishwanath, Durga Temple)
      'historic'→ Historical sites (Ramnagar Fort, Alamgir Mosque)
      'nature'  → Parks and natural areas
      'other'   → Everything else

  - best_season → "October to February" (informational only)

  VALIDATIONS:
  - "required: true" means the field MUST be provided when creating a place
  - "enum: [1, 2, 3]" on priority means ONLY 1, 2, or 3 are accepted
  - "enum: ['dawn', 'morning', 'aarti', 'anytime']" restricts time_slot values
  - If someone tries to save a place with priority: 5, MongoDB rejects it


3. THE ITINERARY MODEL (models/itinerary.model.js)
====================================================

This stores a generated trip plan:

  - user         → Links to the User who created it (MongoDB ObjectId reference)
  - destination  → "Varanasi"
  - days         → 3
  - budget       → 15000
  - groupSize    → 4
  - budgetTier   → 'budget', 'comfort', or 'premium' (auto-detected by engine)
  - plan         → Array of day objects, each containing places

  The plan structure:
  plan: [
    {
      day: 1,
      places: [
        { name: "Kashi Vishwanath", visitDuration: 1.5, entryFee: 0, ... },
        { name: "Dashashwamedh Ghat", visitDuration: 2, entryFee: 0, ... }
      ]
    },
    {
      day: 2,
      places: [
        { name: "Sarnath", visitDuration: 3, entryFee: 50, ... }
      ]
    }
  ]


4. CLOUDINARY — IMAGE STORAGE
===============================

WHAT IS CLOUDINARY?
  A cloud service for storing and serving images. Instead of keeping images
  in our own server or database, we upload them to Cloudinary and store the
  URL in MongoDB.

WHY NOT STORE IMAGES IN MONGODB?
  1. Images are large (1-5 MB each). MongoDB is designed for small documents.
     Storing images would make the database very large and slow.
  2. Cloudinary serves images from a global CDN (Content Delivery Network) —
     servers spread across the world. A user in India gets the image from a
     nearby server, not from our single backend server. Much faster.
  3. Cloudinary can automatically resize, crop, and optimize images.

HOW UPLOAD WORKS (utils/uploadToCloudinary.js):
  1. Admin selects an image file on the Add Place form
  2. Frontend sends the file as "multipart/form-data" (special format for files)
  3. Backend receives the file as a buffer (raw bytes in memory)
  4. We call uploadToCloudinary(buffer) which:
     - Converts the buffer to a base64 string
     - Sends it to Cloudinary's API
     - Cloudinary stores it and returns a URL + publicId
  5. We save the URL in the place document's "image" field
  6. We save the publicId for later deletion

HOW DELETE WORKS (utils/deleteFromCloudinary.js):
  When admin deletes a place:
  1. Find the place → get its publicId
  2. Call deleteFromCloudinary(publicId) → Cloudinary removes the image
  3. Delete the place from MongoDB
  This ensures we don't have orphaned images sitting on Cloudinary forever.


5. PLACE CONTROLLER — CRUD OPERATIONS
=======================================

CRUD = Create, Read, Update, Delete — the four basic operations.

A) CREATE — addPlace (POST /api/place/add)
   Who can do this: Admin only (auth middleware + admin middleware)
   
   Steps:
     1. Extract data from request body (name, city, latitude, etc.)
     2. Check if an image file was attached → if not, return 400 error
     3. Upload image to Cloudinary → get URL and publicId
     4. Create new Place document with all fields
     5. Save to MongoDB
     6. Return 201 with success message and place data

B) READ ALL — getAllPlaces (GET /api/place/all)
   Who can do this: Any logged-in user
   
   Simply: Place.find({}) → returns every place in the database
   The empty {} means "no filter" — get everything.

C) READ ONE — getPlaceById (GET /api/place/:id)
   Who can do this: Any logged-in user
   
   Place.findById(id) → returns the single place with that MongoDB _id
   If not found → return 404 "Place not found"

D) UPDATE — updatePlace (PUT /api/place/:id)
   Who can do this: Admin only
   
   Steps:
     1. Find existing place
     2. If admin uploaded a new image:
        - Upload new image to Cloudinary
        - Delete OLD image from Cloudinary (cleanup)
        - Use new image URL and publicId
     3. If no new image: keep existing image
     4. Update all fields (use new values, or keep old if not provided)
     5. Save changes with findByIdAndUpdate()
        returnDocument: 'after' returns the UPDATED document (not old one)
        runValidators: true ensures the new data passes schema validation

E) DELETE — deletePlace (DELETE /api/place/:id)
   Who can do this: Admin only
   
   Steps:
     1. Find the place (we need the publicId)
     2. Delete image from Cloudinary using publicId
     3. Delete place from MongoDB using findByIdAndDelete(id)
     4. Return success message


6. DASHBOARD CONTROLLER — STATISTICS API
==========================================

A) USER DASHBOARD (GET /api/dashboard/user)

   Returns personal stats for the logged-in user:

   Basic counts:
     - totalItineraries: How many trips they've generated
       → Itinerary.countDocuments({ user: userId })

   Aggregation Pipeline (MongoDB's powerful data processing):
     We use MongoDB's aggregation to calculate multiple stats in ONE query:
     
     $match: Filter only this user's itineraries
     $group: Combine all matching documents into one result:
       - totalBudgetSpent: $sum of all budget values
       - totalPlacesVisited: $sum of array sizes (how many places total)
       - allDestinations: $push all destination names into an array

   Top destinations:
     Group by destination city, count how many times each appears.
     Sort by count descending, limit to top 5.
     Example: [{ _id: "Varanasi", count: 5 }, { _id: "Haridwar", count: 2 }]

   System inventory:
     - Total places in the database (Place.countDocuments())
     - Total unique cities (Place.distinct('city'))

   Recent itineraries:
     The user's 3 most recent trips (sorted by createdAt descending).

B) ADMIN DASHBOARD (GET /api/dashboard/admin)

   Returns system-wide stats:

   Promise.all — runs 6 queries SIMULTANEOUSLY:
     1. Recent itineraries (3 most recent, with user name/email)
     2. Recent places (3 most recently added)
     3. Total user count
     4. Total itinerary count
     5. Total place count
     6. Aggregation with $facet (runs multiple pipelines in one query):
        - Priority distribution: How many Priority 1, 2, 3 sites
        - Budget insights: Average entry fee, most expensive place

   WHY Promise.all?
     Without it: Query 1 runs → waits → Query 2 runs → waits → ...
     Total time: sum of all query times (e.g., 100ms × 6 = 600ms)

     With Promise.all: All 6 queries run at the same time.
     Total time: the slowest query's time (e.g., 100ms)
     About 5-6x faster.

LIKELY VIVA QUESTIONS AND ANSWERS
================================================================================

Q: Why MongoDB instead of MySQL/PostgreSQL?
A: MongoDB stores documents (like JSON objects) which map naturally to our
   JavaScript objects. The itinerary's "plan" field is a nested array of
   objects — this is trivial in MongoDB but would require multiple join
   tables in SQL. Also, MongoDB is schema-flexible, which helped during
   rapid development.

Q: What is Mongoose and why do we need it?
A: Mongoose is an ODM (Object Document Mapper) for MongoDB. It provides
   schemas (data validation), model methods, and query helpers. Without it,
   we'd manually validate every field before saving.

Q: What is an aggregation pipeline?
A: A series of stages that process documents step by step. Like an assembly
   line: $match filters documents → $group combines them → $sort orders them
   → $limit takes the top N. Each stage passes its output to the next.

Q: What is $facet in MongoDB?
A: It lets you run multiple aggregation pipelines on the same data in one
   query. We use it to calculate priority distribution AND budget insights
   in a single database call instead of two.

Q: Why use Cloudinary instead of storing images locally?
A: Three reasons: (1) Images are large, would bloat the database. (2)
   Cloudinary uses a global CDN for fast delivery worldwide. (3) It handles
   image optimization, resizing, and format conversion automatically.

Q: What is multipart/form-data?
A: A special HTTP content type used to send files (images, documents) along
   with text data in a single request. Regular JSON can't contain file data.
   The browser splits the request into "parts" — text fields and file fields.

Q: What does findByIdAndUpdate with returnDocument: 'after' mean?
A: By default, findByIdAndUpdate returns the OLD document (before changes).
   returnDocument: 'after' makes it return the UPDATED document, which is
   what we want to send back to the frontend.

Q: How do you prevent orphaned images in Cloudinary?
A: When updating a place with a new image, we upload the new image FIRST,
   then delete the old one. When deleting a place, we delete the Cloudinary
   image before deleting the database document.

Q: What is the difference between findOne and findById?
A: findById(id) is shorthand for findOne({ _id: id }). Both do the same
   thing, but findById is cleaner for looking up by MongoDB's auto-generated
   _id field.

Q: What does Place.distinct('city') return?
A: An array of unique city names across all places. If we have 50 places
   in Varanasi and 30 in Haridwar, it returns ["Varanasi", "Haridwar"].
